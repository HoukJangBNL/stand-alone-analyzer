# Shared Coding Rules — All Worker Agents

> 이 문서는 `.claude/agents/*.md`에 정의된 모든 워커 에이전트(db-specialist, api-developer, frontend-architect, algo-engineer, devops-engineer, researcher, code-reviewer)에 공통으로 적용되는 코딩 디시플린이다.
> 각 에이전트의 도메인 룰·필수 워크플로는 자기 정의 파일에 따로 적혀 있다 — 이 문서는 그 위에 얹는 **공통 행동 규범**.

각 워커 에이전트는 작업을 시작하기 전 이 파일을 읽고, 모든 변경에 다음 4가지 원칙을 적용한다.

---

## 1. Think Before Coding

코드를 한 줄도 쓰기 전에:

- **가정을 명시한다.** 사용자/PM이 알려준 입력 형태·범위·제약을 본문이나 짧은 코멘트로 적어두고 시작. 가정이 틀렸으면 빨리 들킨다.
- **불확실하면 묻는다.** 한 번에 답할 수 없는 모호한 부분이 있으면, 추측해서 코드를 만들지 않고 PM에게 압축 질문을 던진다 (4지선다 선호).
- **트레이드오프를 표면화한다.** A·B 두 길이 보이면 어느 쪽이 더 단순한지, 어느 쪽이 미래 비용을 키우는지 한 줄로라도 적고 선택. PM이 검수할 때 같은 판단을 재현할 수 있어야 한다.

> **레드 플래그**: "일단 짜고 보자", "되면 되는 대로", "어차피 나중에 리팩터링할 거야" — 전부 이 원칙 위반.

---

## 2. Simplicity First

가장 단순한 동작 코드부터.

- **최소 코드로 요구사항을 충족.** YAGNI. 미래에 쓸지도 모르는 옵션·플래그·추상화 추가 금지.
- **추측 기반 기능 금지.** "이런 케이스도 있을 수 있으니까" 류의 방어 코드는 실제로 들어오는 입력 + 시스템 경계(외부 API·사용자 입력)에서만.
- **시니어가 보면 어떤 반응?** 작성한 코드를 시니어 엔지니어가 본다고 가정. "이거 너무 복잡한데?"라고 할 만한 구간이 있으면 단순화.
- **세 줄 비슷한 코드 > 성급한 추상화.** 중복이 진짜 패턴인지 두 번 더 보고 추상화.

> 워커 에이전트는 자기 도메인의 기존 패턴을 따른다. 기존 코드가 단순한데 자기 PR만 화려하면 그 자체가 오버엔지니어링.

---

## 3. Surgical Changes

요청된 부분만 건드린다.

- **요청에 직접 관련된 코드만 수정.** 사용자/PM이 "X를 고쳐"라고 했는데 PR에 Y·Z 정리까지 들어가면 리뷰가 막힌다. 정리는 별도 태스크로 분리해서 PM에 보고.
- **기존 스타일·패턴을 따른다.** 파일의 import 순서, 네이밍 컨벤션, 에러 처리 방식, 타입 어노테이션 스타일 — 다 맞춘다.
- **변경된 모든 라인이 요청과 연결되어야 한다.** "왜 이 라인이 바뀌었지?"라고 물었을 때 "X 때문"이라고 1초 안에 답할 수 있어야 함. 답할 수 없으면 그 변경은 빼라.
- **무관한 정리·리네이밍·포매팅 변경 금지.** 작업 중 발견한 진짜 문제는 PM에 보고하고 별도 태스크로.

> 코드 리뷰어 에이전트는 이 원칙을 직접 게이트로 사용한다 — surgical scope를 벗어난 변경은 BLOCKER.

---

## 4. Goal-Driven Execution

성공 기준을 먼저 정의하고 시작한다.

- **"끝났다"의 의미를 코드 쓰기 전에 합의.** 어떤 테스트가 통과해야 하는지, 어떤 산출물이 있어야 하는지. PM 브리프에 명시되어 있지 않으면 본인이 쓴 후 진행.
- **테스트로 성공 기준을 표현 (TDD가 필수인 도메인은 그 룰을 따름).** 예: "Add input validation" → 잘못된 입력에 대한 실패 테스트를 먼저 쓰고, 그걸 통과시키는 최소 코드를 작성.
- **불필요한 우회 금지.** 막혔을 때 테스트를 비활성화하거나 검증을 건너뛰어 빨리 "끝낸 척" 하지 않는다. 진짜 막혔으면 PM에 BLOCKED로 에스컬레이션.
- **완료 보고는 검증 가능한 형태로.** "완료" → "테스트 N개 통과, 파일 X·Y·Z 변경, 다음 단계는…" 식. PM이 1차 검증할 수 있어야 한다.

> **레드 플래그**: 막혔다고 테스트를 약화시키기 / `--no-verify` 같은 우회 / "일단 통과시키고 나중에 손보자" — 전부 이 원칙 위반.

---

## 적용 우선순위

이 4가지 룰과 다른 가이드(SuperClaude `~/.claude/RULES.md`, 각 에이전트 정의 파일)가 충돌하면:

1. **사용자/PM의 명시적 지시** — 최우선.
2. **이 파일 + 에이전트 정의의 도메인 룰** — 같은 레이어, 더 구체적인 쪽이 이긴다.
3. **SuperClaude 글로벌 가이드** — 베이스라인.

충돌 자체가 PM에 보고할 신호다.
