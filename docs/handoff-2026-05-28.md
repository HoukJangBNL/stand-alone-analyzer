## 3. 다음 세션의 한 발

**#229 디스패치만 하면 됨.** 사전 결정·자원·코드 다 준비됨.

### 3.1 무엇을 할지
1. `general-purpose` agent에 `.claude/agents/devops-engineer.md` 정의 로드시켜 background dispatch.
2. AMI `ami-092ae5880cb9cf957`로 g6e.48xlarge spot 런치 (LT는 기존 v16, AMI ID만 교체 필요할 수 있음 — 스크립트가 받는지 먼저 확인).
3. 100장 데이터셋(merged_m3 기반)으로 `run_amg_v2.run_multi_process` 실행.
4. **로그 분리 필수**:
   - `boot_s` = 인스턴스 시작 → SSM-online 또는 worker ready
   - `model_load_s` = build_sam2 + merged_m3 로드 시간
   - `processing_s` = 첫 이미지 시작 → 마지막 이미지 끝
   - `total_s` = boot + load + processing
5. 결과 산출: `s_per_card_img`, `scaling_factor_vs_single_gpu` (single GPU baseline 3.98 s/img — `docs/sam-ops.md` §15)
6. 자동 termination, 비용 ~$2 cap, 시간 ~30분 cap.

### 3.2 디스패치 brief 핵심 (가져갈 것)
- AMI ID: `ami-092ae5880cb9cf957`
- Base: AWS DLAMI Ubuntu 22.04 OSS (CUDA 12.9, driver 580.159.04)
- Pre-installed: peft 0.19.1, torch 2.12.0+cu130, peft, vendor submodule `505e1cb2`
- Vendor branch on `HoukJangBNL/QPress-SAM-Flake`: `feat/per-tensor-rank-merge` @ `505e1cb22289317acabc0e466f5f557e9dcd1988`
- Repo branch: `feat/migration-cutover` @ HEAD (commits up to `01ceb7f1`)
- Merged artifact key: `s3://qpress-uploads/internal/sam/merged_m3/sam2.1_hiera_large.merged_m3.3ec586fc.pt` + `.sha256`
- Single-GPU baseline: 3.98 s/img (3648 imgs in ~242 min on g6e.xlarge)
- Multi-GPU baseline (failed 2026-05-28 8-GPU): 12.16 s/card-img (~3.06× slower than single-GPU — what we're trying to fix)

### 3.3 측정 디자인 메모
- `boot_s`와 `processing_s` 분리 안 한 게 #211의 큰 갭. CloudWatch / SSM timestamp / userdata `/var/log/cloud-init.log` markers + worker가 stdin으로 시작/종료 timestamp 찍게 해야 함.
- 100장은 8 GPU면 1 GPU당 12.5장. 8 GPU 분할시 cold start overhead가 dominant 가능 — boot/load 분리가 결정적.
- `merged_m3` 자산이 vendor `run_multi_process` 코드 경로에서 실제 사용되는지 확인 (#209에서 wire 했지만 라이브 검증 안 됨).
