### Task 1: Fix `upsert_from_claims` so email and email_verified_at actually mutate on conflict

**Files:**
- Modify: `src/flake_analysis/api/auth/users.py:36-52`
- Test: `tests/api/test_auth_users_upsert.py` (no edits — drives the fix)

**Failing tests this task unblocks:**
- `tests/api/test_auth_users_upsert.py::test_email_change_updates_row`
- `tests/api/test_auth_users_upsert.py::test_email_unverified_clears_timestamp`

**Root cause:** `on_conflict_do_update`'s `set_={"email": email, "email_verified_at": email_verified_at}` binds Python locals at **statement-build time**. SQLAlchemy renders these as parameters, so the new values *should* land — but `.returning(UserModel.id)` returns only the id, and the subsequent `select(UserModel).where(UserModel.id == user_id)` re-fetches. The actual problem (verified by traceback `assert b.email == 'c@d'` got `'a@b'`) is that `session.commit()` followed by a fresh `select` reads from a session whose identity map still holds the original `User` row. The fix is to **`expire_all()` (or use `expire(user)` + refresh) before re-selecting**, OR to use `RETURNING UserModel` (return all columns) and rely on the upserted row directly.

- [ ] **Step 2: Replace the upsert + refetch with a single RETURNING-all upsert**

In `src/flake_analysis/api/auth/users.py`, replace the body of `upsert_from_claims` (lines 30..61) with:

```python
async def upsert_from_claims(
    session: AsyncSession, claims: dict[str, Any]
) -> UserModel:
    cognito_sub = claims["sub"]
    email = claims.get("email")
    email_verified = claims.get("email_verified", False)
    email_verified_at = datetime.now(timezone.utc) if email_verified else None

    stmt = (
        insert(UserModel)
        .values(
            cognito_sub=cognito_sub,
            email=email,
            email_verified_at=email_verified_at,
            role=UserRole.MEMBER,
        )
        .on_conflict_do_update(
            index_elements=["cognito_sub"],
            set_={
                "email": email,
                "email_verified_at": email_verified_at,
            },
        )
    )
    # Use RETURNING * via the ORM-style returning so we get a populated model.
    stmt = stmt.returning(UserModel)
    result = await session.execute(
        stmt,
        execution_options={"populate_existing": True},
    )
    user = result.scalar_one()
    await session.commit()
    # populate_existing already refreshed the identity-map entry, but if the
    # upsert was an UPDATE the in-memory attributes may be stale. Expire and
    # refresh to read the post-commit state authoritatively.
    await session.refresh(user)
    return user
```

Notes for the implementer:
- Keep the imports already at the top of the file; no new imports needed.
- Do **not** delete the `from sqlalchemy import select` import even if the
  function no longer uses it — leave it if some other helper does; remove
  it only if it becomes the only unused import in the file.
- The `populate_existing=True` execution option is what forces SQLAlchemy
  to overwrite the in-memory row with the upserted column values.

**Root cause:** `fastapi.testclient.TestClient` runs the ASGI app in a
**different event loop** than the asyncpg connection used by the
`pg_session` fixture, so the first DB call produces:

```

```

The fix is to use `httpx.AsyncClient(transport=ASGITransport(app))`,
which runs the app on the **caller's** event loop (the same one
`pytest-asyncio` provides for `pg_session`). `tests/api/test_auth_dep_dropin.py`
already uses this pattern — copy from there.

### Task 4: Auto-seed the dev-bypass user so `emit()` doesn't violate the FK

**Files:**
- Modify: `src/flake_analysis/api/auth/dev_bypass.py` (add an upsert helper)
- Modify: `src/flake_analysis/api/auth/__init__.py:48-52` (call the upsert helper inside the bypass branch)

**Failing tests this task unblocks:**
- `tests/api/test_run_emits_usage.py::test_run_thumbnails_emits_scan_run_event`
- `tests/api/test_run_emits_usage.py::test_run_background_emits_scan_run_event`

**Root cause:** The dev-bypass user has a hard-coded UUID
(`00000000-0000-0000-0000-000000000001`) but no row in `users` table.
`emit()` issues `INSERT INTO usage_events (user_id, ...) VALUES (<dev-uuid>, ...)`
which fails the FK to `users(id)`. We have two options:

1. Wrap `emit()` in `try/except` — silently swallows real bugs in prod.
2. **Seed the dev-bypass user on the first request** — keeps `emit()`
   honest and matches what the prod path does (Cognito callback upserts
   the user before any other request can hit a usage hook).

We pick #2.

- [ ] **Step 1: Add an idempotent upsert helper to `dev_bypass.py`**

Insert below `mint_dev_user`:

```python
from sqlalchemy.dialects.postgresql import insert as pg_insert
from sqlalchemy.ext.asyncio import AsyncSession
from datetime import datetime, timezone

from flake_analysis.db.models.user import User as UserModel


_DEV_USER_UUID = UUID("00000000-0000-0000-0000-000000000001")


async def ensure_dev_user_in_db(session: AsyncSession) -> None:
    """Idempotently insert the dev-bypass user row so FK constraints hold.

    Called from get_current_user when SAA_AUTH_DEV_BYPASS=1. Without this,
    any request that issues a usage_events insert (run/auth hooks) will
    violate the user_id FK because mint_dev_user's UUID has no row.
    """
    stmt = (
        pg_insert(UserModel)
        .values(
            id=_DEV_USER_UUID,
            email="local@dev",
            cognito_sub="dev:local",
            role=UserRole.ADMIN,
            email_verified_at=datetime.now(timezone.utc),
        )
        .on_conflict_do_nothing(index_elements=["id"])
    )
    await session.execute(stmt)
    await session.commit()
```

Update the existing `mint_dev_user` to reuse `_DEV_USER_UUID`:

```python
def mint_dev_user() -> User:
    return User(
        id=_DEV_USER_UUID,
        email="local@dev",
        role=UserRole.ADMIN,
        email_verified=True,
        cognito_sub="dev:local",
    )
```
