# W2.4 — Manifest Endpoint DB Rewire Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Switch `GET /api/v1/projects/{project_id}/data/manifest` from "read on-disk `manifest.json` verbatim" to "merge DB-derived `step_status` (the source of truth post-cutover) onto the on-disk artefact". W4.2 already shipped the pure helper `derive_manifest_steps_from_analysis`; W2.1 already shipped `ProjectContext`. This plan composes the two.

**Architecture:**
- **Add `get_active_analysis()` async dependency.** Resolves `(ProjectContext, async session) → Analysis | None`. v1 selects the most recent `analyses` row whose `scan_id` belongs to the active project. If no row exists (fresh dev instance — currently the case on `local`), returns `None`. The dependency is **read-only** and never inserts.
- **Reshape `get_manifest_endpoint`.** Keep the existing on-disk read as the base. When `Analysis | None` is `Some`, overlay the DB-derived `steps_done` map onto the manifest's `steps` dict using `derive_manifest_steps_from_analysis`. The overlay rule: a DB-truthy step replaces the on-disk entry's `completed_at` semantics (DB wins); steps absent from the DB map are passed through (e.g. `clustering` lives in `manifest.json` only — there is no `analyses.steps_done["clustering"]` slot in v6). Add `status: PipelineStatus` to the response when an analysis exists.
- **No DDL change. No new alembic revision. No write paths added.** This plan is read-side reconciliation.
- **Fallback is silent.** When `Analysis | None` is `None` (no DB row for the project), the endpoint behaves byte-for-byte as today — the migration is invisible to clients that do not yet have a DB-backed project.

**Tech Stack:**
- FastAPI 0.110+, pydantic v2.6+
- SQLAlchemy 2.x async (`AsyncSession`, `select`, `where`, `order_by`)
- pytest 8.x + pytest-asyncio + httpx 0.28.1 `AsyncClient`+`ASGITransport`
- `pytest.mark.pg` opt-in PG fixture (already present per W4.2)

**Pre-read (do not modify):** `docs/superpowers/plans/2026-05-21-W4.2-db-orm.md` (decisions), `src/flake_analysis/db/reconcile.py`, `src/flake_analysis/db/models/analysis.py`, `src/flake_analysis/api/deps.py` (W2.1 wiring), `src/flake_analysis/api/schemas/data.py` (`ManifestModel`).

## Pinned decisions

These pin the design choices needed for W2.4. No later task may walk one back.

1. **DB is the source of truth for `step_status` ONLY for steps in `DB_TO_MANIFEST_STEP_MAP`** — currently `background`, `sam`, `domain_stats`, `domain_proximity`. Anything outside that map (today: `selector`, `clustering`, `thumbnails`, `explorer`) stays disk-driven. Reason: v6 `analyses.steps_done` JSONB is a closed enum and v6 does not have rows for the wrapper-only steps. Touching that requires a new schema rev — out of scope.
2. **`Analysis | None` is the only thing the endpoint resolves from DB.** No `Run`-level join, no audit log fetch, no `metrics` read. Keeping the query narrow keeps the endpoint hot-path-friendly.
3. **"Most recent analysis for the active project" semantics in v1: `ORDER BY analyses.id DESC LIMIT 1` joined to `scans` whose project alias matches `ProjectContext.project_id`.** Until a real `projects` table lands (out of scope), `project_id="local"` matches "any analysis row" — which is exactly what we want during the v1 transition.
4. **No DB write from this endpoint, ever.** The endpoint is a `GET`. If reconciliation later requires *writing* a step status to the DB, that's a separate write path tied to the pipeline runner, not this endpoint.
5. **DB error → 500. NOT silent fallback to disk.** A reachable but failing DB is a real bug; we want the alarm. The "no DB row" case is silent fallback (decision 1's corollary), but a SQL error or session timeout surfaces as a 500 with `error.code="db_unavailable"`.
6. **Test matrix: three cases, one suite.** (a) no DB row → endpoint returns disk manifest unchanged; (b) DB row with `steps_done={"background": true}` → endpoint adds/overrides `steps.background` and exposes `status`; (c) DB session raises → endpoint returns 500 with the documented error envelope. (a) is asyncio-native (no PG required). (b) and (c) live behind `pytest.mark.pg`.
7. **`status` is a NEW top-level field on `ManifestModel`.** Type: `Optional[str]` (the lowercase enum name). When DB row absent → `None`. This is additive; existing clients ignore unknown fields.
8. **No frontend change in this plan.** `web/src/api/manifest.ts` will pick up the new `status` field automatically once the backend ships; making it visible in the UI is a follow-up for the project status banner (out of scope here).
9. **TDD: red→green per task.** Disk-only fallback (case a) is the red test for Task 2. PG-backed cases (b, c) are red for Task 3.

# src/flake_analysis/api/services/manifest_merge.py
"""Merge DB-derived step status into a manifest-shaped dict.

Source of truth: ``analyses.steps_done`` for steps in
``DB_TO_MANIFEST_STEP_MAP``. Other steps stay disk-driven.
"""
from __future__ import annotations
from typing import TYPE_CHECKING, Any

from flake_analysis.db.reconcile import (
    analysis_status_string,
    derive_manifest_steps_from_analysis,
)

if TYPE_CHECKING:
    from flake_analysis.db.models import Analysis


def merge_db_steps_into_manifest(
    manifest_dict: dict[str, Any],
    analysis: "Analysis | None",
) -> dict[str, Any]:
    """Overlay DB-derived steps onto a manifest dict and add `status`.

    Returns the same dict object when ``analysis is None`` (no defensive
    copy — caller already owns it). Mutates the dict otherwise.
    """
    if analysis is None:
        return manifest_dict
    db_steps = derive_manifest_steps_from_analysis(analysis)
    steps_out = dict(manifest_dict.get("steps") or {})
    steps_out.update(db_steps)  # DB wins for keys it covers.
    manifest_dict["steps"] = steps_out
    manifest_dict["status"] = analysis_status_string(analysis)
    return manifest_dict
```

# src/flake_analysis/api/deps.py — append
from typing import Annotated, AsyncIterator
from fastapi import Depends
from sqlalchemy import select
from sqlalchemy.ext.asyncio import AsyncSession

from flake_analysis.db.engine import async_session_maker
from flake_analysis.db.models import Analysis


async def get_db_session() -> AsyncIterator[AsyncSession]:
    """Yield an async session per request; close on exit."""
    async with async_session_maker() as session:
        yield session


async def get_active_analysis(
    ctx: Annotated[ProjectContext, Depends(get_project_context)],
    session: Annotated[AsyncSession, Depends(get_db_session)],
) -> Analysis | None:
    """Resolve the active Analysis row for the request's ProjectContext.

    v1 selects the most recent analyses row joined to scans whose project
    alias matches ctx.project_id. v1 always uses 'local' so this devolves
    to ``ORDER BY analyses.id DESC LIMIT 1`` — that is the v1 contract.
    """
    stmt = select(Analysis).order_by(Analysis.id.desc()).limit(1)
    result = await session.execute(stmt)
    return result.scalar_one_or_none()
```
