# W6.6 T5 — Test B Final Acceptance: get_db_session Override + Role-Mismatch Fix

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Bring W6 Test B (local PG integration) from 38 passed / 9 failed → 47 passed / 0 failed by overriding `get_db_session` in PG-marked API tests so the FastAPI route shares the test's `pg_session` (and its rollback-only transaction), and rewriting the one role-mismatch test to override `get_current_user` instead of relying on dev-bypass that always returns ADMIN.

**Architecture:** All 9 failures share a single root cause: routes acquire a fresh connection via `get_db_session()` (which builds a session from `async_session_maker` in `flake_analysis.db.engine`), so (a) the writes the test made on `pg_session` are invisible to the route, (b) reads back through `pg_session` after the route can't see what the route wrote (the route committed on a different transaction in a different connection), and (c) the default DB URL points to `qpress`, which doesn't exist on the local dev box — so the route's connection actually errors out before doing anything. Fix: in PG tests, override `app.dependency_overrides[get_db_session]` to yield the test's `pg_session` for the duration of the request. The role-mismatch test can't co-exist with dev-bypass (which hard-codes ADMIN); we override `get_current_user` directly to return a domain MEMBER user.

**Tech Stack:** pytest-asyncio strict, httpx AsyncClient + ASGITransport, FastAPI dependency_overrides, SQLAlchemy 2.x async.

## Task 1: Override get_db_session in admin usage tests + rework role test

**Files:**
- Modify: `tests/api/test_admin_usage_route.py`

**Why:** 7 tests fail. 6 of them write events on `pg_session` then expect to see them via the route, but the route reads via a separate connection on the production engine — which doesn't even exist locally (DB `qpress` not present). 1 test (`test_admin_usage_requires_admin_role`) creates a MEMBER user via `sample_user_factory` and expects 403, but dev-bypass short-circuits `get_current_user` to a hard-coded ADMIN, so the route returns 200.

### Step 1.1: Write the override helper inline + apply to all "happy path" tests

Replace the body of each of these 6 tests so the route shares `pg_session`:
- `test_admin_usage_returns_all_events`
- `test_admin_usage_filters_by_user_id`
- `test_admin_usage_filters_by_kind`
- `test_admin_usage_filters_by_time_range`
- `test_admin_usage_aggregate_mode`
- `test_admin_usage_respects_limit`

**Pattern to apply at the start of each happy-path test (before the `AsyncClient` block):**

```python
from flake_analysis.api.deps import get_db_session

async def _yield_pg_session():
    yield pg_session

app.dependency_overrides[get_db_session] = _yield_pg_session
try:
    # ... existing body that opens AsyncClient and asserts ...
finally:
    app.dependency_overrides.pop(get_db_session, None)
```

**Critical detail:** the override factory must be `async def` and must `yield` (not `return`) the session so FastAPI's `Depends(get_db_session)` machinery treats it as an async generator dependency, matching the production signature.

**Also:** keep `await pg_session.commit()` after the `emit` calls in each test. Even though `pg_session` is rolled back at teardown, the route's read inside the same connection will see the committed transaction (FastAPI is now reading from the same connection).

Wait — actually, the conftest wraps `pg_session` in an outer transaction via `await conn.begin()`, and `session.commit()` inside that bound session emits a SAVEPOINT release, not a real commit. The outer transaction is rolled back at teardown. So writes are visible to the same connection (and to anything that uses the same session) for the duration of the test. That's what we want.

### Step 1.2: Rewrite `test_admin_usage_requires_admin_role`

Replace its body to override `get_current_user` instead of using dev-bypass + sample_user_factory:

```python
@pytest.mark.asyncio
async def test_admin_usage_requires_admin_role(monkeypatch, pg_session):
    """GET /admin/usage returns 403 for non-admin users."""
    from uuid import uuid4
    from flake_analysis.api.auth import User, get_current_user
    from flake_analysis.api.deps import get_db_session
    from flake_analysis.db.models import UserRole

    # Don't use dev-bypass for this test — it hard-codes ADMIN.
    monkeypatch.setenv("SAA_AUTH_DEV_BYPASS", "0")

    member = User(
        id=uuid4(),
        email="member@test.com",
        cognito_sub="member-sub",
        role=UserRole.MEMBER,
        email_verified=True,
    )

    async def _yield_pg_session():
        yield pg_session

    app.dependency_overrides[get_db_session] = _yield_pg_session
    app.dependency_overrides[get_current_user] = lambda: member
    try:
        async with AsyncClient(
            transport=ASGITransport(app=app), base_url="http://t"
        ) as c:
            r = await c.get("/api/v1/admin/usage")
            assert r.status_code == 403
    finally:
        app.dependency_overrides.pop(get_db_session, None)
        app.dependency_overrides.pop(get_current_user, None)
```

**Why no `sample_user_factory`:** the route doesn't read any user row to make the role decision — `require_role` reads `current_user.role` from the dependency's return value. So we don't need a DB row at all for this test.

**Why no `await sample_user_factory(...)`:** removing it eliminates a confounding variable (we want exactly one reason this returns 403: the role).

## Task 3: Fix `test_auth_callback_exchanges_code` — same override

**Files:**
- Modify: `tests/api/test_auth_routes.py`

**Why:** This test passes in isolation (PASSED when run alone) but fails when run with siblings. The error reported in the prior session was `RuntimeError: Event loop is closed` from asyncpg cleanup. Investigation: the test's `monkeypatch.setattr(httpx, "AsyncClient", lambda: mock_client)` replaces the global symbol; if the route's `httpx.AsyncClient` resolves through the patched module, the mock is used. But the route ALSO calls `upsert_from_claims(session, ...)` and `emit(session, ...)` — both committing on a session built by `get_db_session`, which opens a real asyncpg connection on the production engine. When the test's event loop ends, that connection's cleanup races with the next test's loop. Root cause: same engine mismatch.

Apply the same `get_db_session` override and the symptoms disappear. The route will use `pg_session`, no extra asyncpg connection is opened on the production engine, no cross-test loop contamination.

### Step 3.1: Apply override

In `test_auth_callback_exchanges_code` (only this one test in the file is affected; the other tests pass), add the override pattern around the `AsyncClient` block exactly like Task 2.

```python
from flake_analysis.api.deps import get_db_session

async def _yield_pg_session():

app.dependency_overrides[get_db_session] = _yield_pg_session
try:

        r = await c.post(
            "/api/v1/auth/callback",
            json={
                "code": "test-auth-code",
                "redirect_uri": "http://localhost:5173/auth/callback",
            },
        )
```
